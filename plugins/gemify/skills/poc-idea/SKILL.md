# PoC Idea Skill

아이디어 수준의 모호한 요청을 **본질 질문**으로 정제합니다.

## 언제 사용

- "이런 거 만들고 싶어" - 아이디어 수준
- 구체적인 화면/기능 설명 없음
- 문제는 느껴지지만 해결책이 모호함

## 핵심 역할

### 본질 질문 3가지

사용자가 진짜 원하는 것을 끌어내는 질문:

1. **"진짜 해결하려는 문제가 뭐야?"**
   - 표면적 요구 뒤에 숨은 근본 문제
   - "~하고 싶다" → "왜 그게 필요해?"

2. **"그게 되면 뭐가 달라져?"**
   - 기대하는 변화/결과
   - 성공 상태의 구체적 모습

3. **"지금은 어떻게 하고 있어? 뭐가 불편해?"**
   - 현재 상태와 pain point
   - 기존 해결책의 한계

### 부차적인 것 잘라내기

- "그건 나중에, 지금은 X에 집중하자"
- 핵심 문제 1개로 수렴
- 여러 문제가 섞여 있으면 분리

## 워크플로우

```
사용자 입력 (모호한 아이디어)
    ↓
본질 질문 1: "진짜 해결하려는 문제가 뭐야?"
    ↓
본질 질문 2: "그게 되면 뭐가 달라져?"
    ↓
본질 질문 3: "지금은 어떻게 해? 뭐가 불편해?"
    ↓
핵심 문제 정리 + 사용자 확인
    ↓
inbox/thoughts에 저장
    ↓
다음 단계 안내 (poc-shape)
```

## 대화 예시

```
사용자: "gemify를 웹앱으로 만들고 싶어"

Claude: 웹앱으로 뭘 해결하려고 해요?
        (CLI로는 안 되는 게 있나요?)

사용자: "생각 정리를 더 쉽게 하고 싶어"

Claude: 지금은 어떻게 하고 있어요? 뭐가 불편해요?

사용자: "claude code에서 플러그인으로 하는데,
        명령어 치기 귀찮고 시각적으로 안 보여"

Claude: 그게 해결되면 뭐가 달라질 것 같아요?

사용자: "생각의 흐름을 한눈에 보면서
        어디서 막혔는지 파악하고 싶어"

Claude: 정리하면:
        - 문제: CLI에서 지식 흐름을 시각적으로 파악하기 어려움
        - 기대: 흐름을 보면서 막힌 곳 발견
        - 핵심: 시각적 지식 파이프라인

        이렇게 정리해서 inbox에 저장할까요?
```

## 결과물

### inbox/thoughts 저장 형식

```markdown
---
type: poc-idea
created: YYYY-MM-DD
status: idea
---

# {아이디어 제목}

## 문제
{진짜 해결하려는 문제}

## 기대 결과
{그게 되면 달라지는 것}

## 현재 상태
{지금 하고 있는 방식과 불편함}

## 핵심 한 줄
{문제를 한 문장으로}
```

### 파일명 규칙
```
inbox/thoughts/YYYY-MM-DD-{slug}.md
```

예: `inbox/thoughts/2026-01-18-gemify-visual-pipeline.md`

## 완료 후 안내

```
아이디어가 inbox에 저장되었습니다:
  ~/.gemify/inbox/thoughts/2026-01-18-gemify-visual-pipeline.md

핵심 문제: 시각적 지식 파이프라인

다음 단계:
  형태를 구체화하려면 → /gemify:poc (poc-shape로 라우팅)
  바로 만들어보려면 → /gemify:poc "화면 만들어봐" (poc-proto로 라우팅)
```

## 규칙

- **질문은 3개 이내**: 길어지면 핵심을 놓침
- **한 번에 하나**: 여러 문제가 있으면 하나씩 분리
- **저장 전 확인**: 정리 내용 사용자 동의 후 저장
- **부차적 요구 잘라내기**: "그건 나중에" 적극 활용
