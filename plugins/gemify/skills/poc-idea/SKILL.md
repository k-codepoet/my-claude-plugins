# PoC Idea Skill

아이디어 수준의 모호한 요청을 **본질 질문**으로 정제하고, **고객 시나리오**로 밀도를 높입니다.

## 언제 사용

- "이런 거 만들고 싶어" - 아이디어 수준
- 구체적인 화면/기능 설명 없음
- 문제는 느껴지지만 해결책이 모호함

## 핵심 원칙

> "제품을 만들기로 했으면, 코드보다 고객 시나리오를 먼저 작성하라."

시나리오 없이 바로 PoC를 만들면:
- "누구를 위한 제품인지" 불명확
- 기능 나열형 문서가 됨
- MVP 범위가 넓어지고 밀도가 낮아짐

## 워크플로우

```
사용자 입력 (모호한 아이디어)
    ↓
[1단계] 본질 질문
    ├─ "진짜 해결하려는 문제가 뭐야?"
    ├─ "그게 되면 뭐가 달라져?"
    └─ "지금은 어떻게 해? 뭐가 불편해?"
    ↓
핵심 문제 정리 + 사용자 확인
    ↓
[2단계] 고객 시나리오 (필수)
    ├─ 페르소나 정의
    ├─ Pain Points (3가지)
    └─ 사용 시나리오 (대화 형식)
    ↓
시나리오 정리 + 사용자 확인
    ↓
inbox/thoughts에 저장
    ↓
다음 단계 안내 (poc-shape)
```

## 1단계: 본질 질문

### 본질 질문 3가지

사용자가 진짜 원하는 것을 끌어내는 질문:

1. **"진짜 해결하려는 문제가 뭐야?"**
   - 표면적 요구 뒤에 숨은 근본 문제
   - "~하고 싶다" → "왜 그게 필요해?"

2. **"그게 되면 뭐가 달라져?"**
   - 기대하는 변화/결과
   - 성공 상태의 구체적 모습

3. **"지금은 어떻게 하고 있어? 뭐가 불편해?"**
   - 현재 상태와 pain point
   - 기존 해결책의 한계

### 부차적인 것 잘라내기

- "그건 나중에, 지금은 X에 집중하자"
- 핵심 문제 1개로 수렴
- 여러 문제가 섞여 있으면 분리

## 2단계: 고객 시나리오 (필수)

본질 질문으로 문제를 정제한 후, **반드시** 고객 시나리오를 작성합니다.

### 시나리오 작성 프롬프트

```
Claude: "이 제품을 쓸 사람은 누구인가요?
         - 역할 (예: 개발자, PM, 직장인)
         - 특징 (예: 아이디어가 많음, 바쁨)
         - 핵심 니즈 (예: 생각 정리, 자기 발견)"

Claude: "그 사람이 현재 겪는 불편함 3가지는?"

Claude: "이 제품을 쓰는 장면을 대화로 그려볼까요?
         (예: 사용자가 뭐라고 하면, 제품이 어떻게 반응하는지)"
```

### 시나리오 구성 요소

| 항목 | 설명 | 예시 |
|------|------|------|
| **페르소나** | 이름 + 역할 + 특징 + 니즈 | Spark, 기획자, 아이디어가 많음, 포착 필요 |
| **Pain Points** | 현재 겪는 불편함 3가지 | 아이디어 휘발, 연결 안됨, 실행 안됨 |
| **사용 시나리오** | 대화 형식으로 구체화 | "Spark: 이거 저장해줘" → "Gemify: inbox에 저장했어요" |
| **핵심 가치** | 이 사람에게 제품이 주는 것 | 아이디어 휘발 방지 |

## 대화 예시

```
사용자: "gemify를 웹앱으로 만들고 싶어"

Claude: 웹앱으로 뭘 해결하려고 해요?
        (CLI로는 안 되는 게 있나요?)

사용자: "생각 정리를 더 쉽게 하고 싶어"

Claude: 지금은 어떻게 하고 있어요? 뭐가 불편해요?

사용자: "claude code에서 플러그인으로 하는데,
        명령어 치기 귀찮고 시각적으로 안 보여"

Claude: 그게 해결되면 뭐가 달라질 것 같아요?

사용자: "생각의 흐름을 한눈에 보면서
        어디서 막혔는지 파악하고 싶어"

Claude: 좋아요, 핵심 문제가 정리됐네요:
        - 문제: CLI에서 지식 흐름을 시각적으로 파악하기 어려움
        - 기대: 흐름을 보면서 막힌 곳 발견

        이제 이 제품을 쓸 사람을 구체화해볼게요.
        이 웹앱을 쓸 사람은 누구인가요?
        - 역할은? (개발자, PM, 직장인 등)
        - 특징은? (바쁨, 아이디어 많음 등)

사용자: "1인 개발자, 사이드 프로젝트 많이 하는 사람"

Claude: 그 사람이 현재 겪는 불편함 3가지는 뭘까요?

사용자: "CLI 명령어 기억 안남, 전체 흐름 파악 어려움,
        어디서 막혔는지 모름"

Claude: 이 사람이 웹앱을 쓰는 장면을 대화로 그려볼까요?

사용자: "대시보드 열면 파이프라인이 보이고,
        막힌 곳 클릭하면 다음 액션 추천해주는 느낌"

Claude: 정리할게요:

        **페르소나**
        - 이름: Buildy (1인 개발자)
        - 특징: 사이드 프로젝트 많음, CLI 익숙하지만 시각화 원함
        - 니즈: 지식 흐름 한눈에 파악

        **Pain Points**
        1. CLI 명령어 기억 안남
        2. 전체 흐름 파악 어려움
        3. 어디서 막혔는지 모름

        **사용 시나리오**
        Buildy: (대시보드 접속)
        Gemify: 파이프라인 시각화 표시
                └─ inbox(3) → drafts(1) → library(12)
                └─ 막힘 표시: drafts에 1주일 된 항목
        Buildy: (막힌 곳 클릭)
        Gemify: "이 draft를 library로 옮길까요?
                 또는 더 다듬을까요?"

        **핵심 가치**: 지식 흐름 시각화로 막힌 곳 발견

        이렇게 저장할까요?
```

## 결과물

### inbox/thoughts 저장 형식

```markdown
---
type: poc-idea
created: YYYY-MM-DD
status: idea
---

# {아이디어 제목}

## 문제
{진짜 해결하려는 문제}

## 기대 결과
{그게 되면 달라지는 것}

## 현재 상태
{지금 하고 있는 방식과 불편함}

## 고객 시나리오

### 페르소나
- **이름**: {이름}
- **역할**: {역할}
- **특징**: {특징}
- **니즈**: {핵심 니즈}

### Pain Points
1. {불편함 1}
2. {불편함 2}
3. {불편함 3}

### 사용 시나리오
```
{사용자}: {행동/말}
{제품}: {반응}
...
```

### 핵심 가치
{이 사람에게 제품이 주는 것 한 문장}

## 핵심 한 줄
{문제를 한 문장으로}
```

### 파일명 규칙
```
inbox/thoughts/YYYY-MM-DD-{slug}.md
```

예: `inbox/thoughts/2026-01-18-gemify-visual-pipeline.md`

## 완료 후 안내

```
아이디어가 inbox에 저장되었습니다:
  {domain_path}/inbox/thoughts/2026-01-18-gemify-visual-pipeline.md

페르소나: Buildy (1인 개발자)
핵심 문제: 지식 흐름 시각화로 막힌 곳 발견

다음 단계:
  형태를 구체화하려면 → /gemify:poc (poc-shape로 라우팅)
  바로 만들어보려면 → /gemify:poc "화면 만들어봐" (poc-proto로 라우팅)
```

## 규칙

- **본질 질문 3개 이내**: 길어지면 핵심을 놓침
- **시나리오 필수**: 본질 질문 후 반드시 고객 시나리오 작성
- **한 번에 하나**: 여러 문제가 있으면 하나씩 분리
- **저장 전 확인**: 정리 내용 사용자 동의 후 저장
- **부차적 요구 잘라내기**: "그건 나중에" 적극 활용
- **대화 형식 시나리오**: 추상적 설명 대신 구체적 대화로
