# PoC Shape Skill

아이디어를 **구체적인 형태**로 다듬습니다. 현재 도메인의 지식 체계를 활용하여 재료를 찾고, 무의식 도출 질문으로 사용자가 진짜 원하는 모습을 끌어냅니다.

**반드시 `skills/scope/SKILL.md` 참조하여 현재 도메인 경로 결정.**

## 언제 사용

- 핵심 문제는 정의됨
- 하지만 화면/기능이 구체적이지 않음
- inbox에 관련 아이디어가 있음
- 현재 도메인 지식 체계에서 재료 탐색이 도움될 것

## 핵심 역할

### 1. 현재 도메인 지식 체계 탐색

아이디어 실행에 필요한 관련 재료 검색:

```
{domain_path}/ 탐색
├── library/     → 관련 원칙/스펙/how-to
├── drafts/      → 진행 중인 관련 생각
└── views/       → 유사한 이전 작업
```

**탐색 방법:**
- 아이디어 키워드로 grep
- 관련 도메인 폴더 확인
- 최근 수정된 관련 파일 우선

### 2. 재료 추천 조합

```
[재료 발견]
library/principles/user-first.md에서:
  "복잡한 기능보다 핵심 하나를 완벽하게"

views/by-poc/session-viewer.md에서:
  "채팅 + 시각화 분할 레이아웃"

이런 재료들이 있는데, 이렇게 조합하면 어떨까요?
```

### 3. 무의식 도출 질문

사용자의 설계도가 명확해질 때까지 반복:

1. **"화면에 뭐가 보여야 해?"**
   - 가장 먼저 보고 싶은 한 화면
   - 핵심 UI 요소

2. **"그게 없으면 뭐가 불편해?"**
   - 필수 vs 부차적 구분
   - 최소 MVP 범위

3. **"가장 먼저 해결해야 할 한 가지는?"**
   - 우선순위 결정
   - 집중 포인트

## 워크플로우

```
입력: poc-idea에서 생성된 아이디어 또는 사용자 설명
    ↓
{domain_path} 탐색 → 관련 재료 3개 이상 찾기
    ↓
재료 추천: "이런 재료가 있는데, 이렇게 조합하면?"
    ↓
무의식 도출 질문 1: "화면에 뭐가 보여야 해?"
    ↓
무의식 도출 질문 2: "그게 없으면 뭐가 불편해?"
    ↓
무의식 도출 질문 3: "가장 먼저 해결해야 할 한 가지는?"
    ↓
(반복: 형태가 명확해질 때까지)
    ↓
형태 문서 정리 + 사용자 확인
    ↓
drafts에 저장 (sources로 사용된 재료 연결)
```

## 대화 예시

```
사용자: "inbox에 저장한 gemify 웹앱 아이디어를 구체화하고 싶어"

Claude: 현재 도메인의 지식 체계를 탐색해볼게요...

[재료 발견]
- library/specs/gemify-pipeline.md: inbox→draft→library 흐름
- views/by-poc/session-viewer.md: 채팅+시각화 레이아웃
- drafts/visual-thinking.md: 시각적 사고 도구 아이디어

이 재료들로 조합하면:
채팅 인터페이스 + 파이프라인 시각화 + 실시간 상태 표시

어떻게 생각해요?

사용자: "좋은데, 파이프라인 시각화가 더 중요해"

Claude: 화면에 뭐가 보여야 해요?
        파이프라인 시각화라면, 어떤 형태?

사용자: "inbox → draft → library 흐름이 보이고,
        각 단계에 뭐가 있는지 한눈에"

Claude: 그게 없으면 뭐가 불편해요?

사용자: "지금 어디서 막혔는지 모르겠어.
        생각이 어디까지 다듬어졌는지도."

Claude: 가장 먼저 보고 싶은 한 화면은?

사용자: "파이프라인 전체 뷰. 각 단계 클릭하면 목록 나오게"

Claude: 정리하면:
        - 핵심 화면: 파이프라인 전체 뷰 (inbox→draft→library)
        - 기능: 단계 클릭 → 해당 단계 문서 목록
        - 해결: 진행 상태 시각화로 막힌 곳 파악

        이렇게 drafts에 저장할까요?
```

## 결과물

### drafts 저장 형식

```markdown
---
type: poc-shape
created: YYYY-MM-DD
status: shaping
sources:
  - library/specs/gemify-pipeline.md
  - views/by-poc/session-viewer.md
---

# {프로젝트명} 형태

## 핵심 화면

### 1. {화면명}
- 목적: {이 화면이 해결하는 것}
- 구성요소:
  - {요소 1}
  - {요소 2}

## MVP 범위
- 필수: {반드시 있어야 할 것}
- 부차적: {나중에 해도 되는 것}

## 사용 시나리오
1. 사용자가 {행동}
2. 시스템이 {반응}
3. 결과로 {상태 변화}

## 참고 재료
- {사용된 library/drafts/views 파일들}
```

### 파일명 규칙
```
drafts/YYYY-MM-DD-{slug}-shape.md
```

예: `drafts/2026-01-18-gemify-webapp-shape.md`

## 완료 후 안내

```
형태 문서가 drafts에 저장되었습니다:
  {domain_path}/drafts/2026-01-18-gemify-webapp-shape.md

핵심 화면: 파이프라인 전체 뷰

다음 단계:
  화면 만들어보려면 → /gemify:poc "파이프라인 뷰 만들어봐"
  (poc-proto가 HTML로 바로 생성 + 미리보기)
```

## 규칙

- **재료 3개 이상**: 현재 도메인에서 최소 3개 관련 재료 찾기
- **질문은 구체적으로**: "어떻게?" 보다 "뭐가 보여야 해?"
- **sources 연결**: 사용된 재료를 frontmatter에 기록
- **shape ↔ proto 반복 가능**: 형태 확인 후 다시 다듬기 OK
